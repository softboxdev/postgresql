# Организация данных в PostgreSQL: логическая и физическая структура

## Введение

PostgreSQL — это сложная система, которая скрывает от пользователя детали того, *как именно* данные хранятся на диске. Вы пишете простой SQL, а внутри происходит магия преобразования "строк и таблиц" в "байты на диске".

Этот разбор откроет "второй слой": как абстрактная `таблица` превращается в файлы на файловой системе.

---

## Часть 1. Логическая структура (То, что видит пользователь)

Логическая структура — это то, с чем вы работаете через SQL. Это **абстракция**, которая скрывает сложность.

### 1.1. Иерархия логических объектов

```
Cluster (Кластер БД)
   │
   ├── Database 1 (БД "shop")
   │      │
   │      ├── Schema "public"
   │      │      │
   │      │      ├── Table "users"
   │      │      │      ├── Column "id" (INT)
   │      │      │      ├── Column "name" (VARCHAR)
   │      │      │      └── Constraint "PRIMARY KEY"
   │      │      │
   │      │      ├── Index "idx_users_name"
   │      │      └── View "active_users"
   │      │
   │      └── Schema "audit"
   │
   └── Database 2 (БД "blog")
```

### 1.2. Что такое каждый уровень логической структуры?

| Уровень | Назначение | Пример |
|---------|------------|--------|
| **Cluster** | Группа всех БД на одном экземпляре PostgreSQL | Процесс `postmaster`, порт 5432, папка `/var/lib/postgresql/data` |
| **Database** | Изолированное хранилище данных (свои таблицы, пользователи) | `shop_db`, `blog_db` |
| **Schema** | Пространство имён внутри БД (логическая группировка) | `public` (по умолчанию), `sales`, `hr` |
| **Table** | Набор строк с одинаковой структурой (колонками) | `users`, `orders` |
| **Column** | Атомарный элемент данных с определённым типом | `name VARCHAR(100)`, `created_at TIMESTAMP` |
| **Row** | Одна запись (набор значений всех колонок) | `(1, 'Иван', 'ivan@mail.ru')` |

### 1.3. Логические связи между таблицами

```sql
-- Связь "один ко многим" (1:N)
CREATE TABLE users (
    id SERIAL PRIMARY KEY   -- Один пользователь
);

CREATE TABLE orders (
    id SERIAL PRIMARY KEY,
    user_id INTEGER REFERENCES users(id)  -- Много заказов
);

-- Связь "многие ко многим" (M:N) через промежуточную таблицу
CREATE TABLE books (
    id SERIAL PRIMARY KEY
);

CREATE TABLE authors (
    id SERIAL PRIMARY KEY
);

CREATE TABLE book_authors (
    book_id INTEGER REFERENCES books(id),
    author_id INTEGER REFERENCES authors(id)
);
```

### 1.4. Логические ограничения (Constraints)

| Ограничение | Что проверяет | Пример |
|-------------|---------------|--------|
| `NOT NULL` | Значение обязательно | `name VARCHAR NOT NULL` |
| `UNIQUE` | Значение уникально в таблице | `email VARCHAR UNIQUE` |
| `PRIMARY KEY` | `NOT NULL` + `UNIQUE` | `id SERIAL PRIMARY KEY` |
| `FOREIGN KEY` | Ссылка на существующую запись | `user_id REFERENCES users(id)` |
| `CHECK` | Пользовательское условие | `age INTEGER CHECK (age >= 18)` |

## Часть 2. Физическая структура (Как это хранится на диске)

Это самое важное для понимания производительности. За красивым интерфейсом SQL скрывается жёсткая файловая реальность.

### 2.1. Общая схема физического хранения

```
                         PostgreSQL Cluster
                               │
        ┌──────────────────────┼──────────────────────┐
        │                      │                      │
   Database 1              Database 2              Database 3
   (каталог)              (каталог)               (каталог)
        │
        ├───────────────────┼───────────────────┐
        │                   │                   │
   Таблица "users"       Таблица "orders"     Индекс "idx_..."
   (файл: 16384)         (файл: 16385)        (файл: 16386)
        │
        ├───────────────────┼───────────────────┐
        │                   │                   │
   Файл главных          Файл карты          Файл "заморозки"
   данных (.1)           видимости (.fsm)    (vm)
```

### 2.2. Где лежат файлы физически (в ОС)

По умолчанию данные хранятся в:
```bash
/var/lib/postgresql/16/main/base/
```

Посмотрим на реальные файлы:
```bash
sudo -u postgres ls -la /var/lib/postgresql/16/main/base/
```

**Пример вывода:**
```
drwx------ 2 postgres postgres 4096 Jan 15 10:00 16384   # <- База данных 1 (OID=16384)
drwx------ 2 postgres postgres 4096 Jan 15 10:00 16385   # <- База данных 2
-rw------- 1 postgres postgres 8192 Jan 15 10:05 16386   # <- Таблица или индекс
```

### 2.3. Отображение логики на физику: OID и RelFileNode

Каждый объект в PostgreSQL имеет числовой идентификатор — **OID** (Object IDentifier).

```sql
-- Посмотреть OID базы данных
SELECT oid, datname FROM pg_database;

-- Посмотреть OID таблицы и её физический файл
SELECT relname, relfilenode, relpages, reltuples 
FROM pg_class 
WHERE relname = 'users';
```

**Пример:**
```
 relname | relfilenode | relpages | reltuples 
---------+-------------+----------+-----------
 users   |       16386 |      100 |     10000
```

**Что это значит:**
- Таблица `users` физически лежит в файле с номером `16386`
- Занимает `100` страниц на диске (100 × 8 КБ = 800 КБ)
- Содержит примерно `10000` строк

### 2.4. Физический файл таблицы (Heap File)

```
Файл: base/16384/16386
┌─────────────────────────────────────────────────────────────┐

│ - pd_upper: 7840 (конец данных)                              │
│ - pd_special: 8192                                           │

│ Line Pointer 1: смещение 48, длина 128 (указывает на строку) │

│ ├─ t_xmax: 0 (ID транзакции, которая удалила)               │

                            ...
```

### 2.5. Детальный разбор страницы (Page)

Каждая страница — это 8 КБ (8192 байта). Формат фиксированный, но содержимое меняется.

#### Заголовок страницы (PageHeaderData) — 24 байта

```c
typedef struct PageHeaderData
{
    PageXLogRecPtr pd_lsn;   // 8 байт: позиция в WAL (журнале)
    uint16        pd_checksum; // 2 байта: контрольная сумма
    uint16        pd_flags;    // 2 байта: флаги (содержит ли строки и т.д.)
    LocationIndex pd_lower;    // 2 байта: смещение начала данных (обычно 48)
    LocationIndex pd_upper;    // 2 байта: смещение конца данных (свободное место)
    LocationIndex pd_special;  // 2 байта: смещение специальной области (для индексов)
    uint16        pd_pagesize_version; // 2 байта: версия формата
    TransactionId pd_prune_xid; // 4 байта: для очистки (VACUUM)
} PageHeaderData;
```

#### Line Pointer (указатель на строку) — 4 байта на строку

```c
typedef struct ItemIdData
{
    unsigned lp_off:15,    // смещение от начала страницы (15 бит)
             lp_flags:2,   // флаги (используется, удалена и т.д.)
             lp_len:15;    // длина строки (15 бит)
} ItemIdData;
```

**Как это работает:**
1. Когда вставляется новая строка, PostgreSQL выделяет место в конце страницы (от `pd_upper` вниз)
2. Добавляет указатель в начало (от `pd_lower` вверх)
3. `pd_lower` увеличивается на 4 байта (новый указатель)
4. `pd_upper` уменьшается на длину строки

### 2.6. Структура HeapTuple (строки данных) — Заголовок 23 байта + данные

```c
typedef struct HeapTupleHeaderData
{
    union {
        TransactionId t_xmin;    // 4 байта: ID транзакции, ВСТАВИВШЕЙ строку
        TransactionId t_xvac;    // 4 байта: для VACUUM
    } t_choice;
    union {
        TransactionId t_xmax;    // 4 байта: ID транзакции, УДАЛИВШЕЙ строку
        CommandId    t_cid;      // 4 байта: команда внутри транзакции
    } t_field3;
    
    ItemPointerData t_ctid;      // 6 байт: физический адрес строки (страница, смещение)
    uint16         t_infomask2;  // 2 байта: флаги (количество атрибутов и т.д.)
    uint16         t_infomask;   // 2 байта: флаги (была ли строка изменена, имеет ли NULL)
    uint8          t_hoff;       // 1 байт: смещение до данных (обычно 32)
    
    // ДАННЫЕ НАЧИНАЮТСЯ ЗДЕСЬ
    // ... битовая карта NULL ...
    // ... значения колонок ...
} HeapTupleHeaderData;
```

**Пример заголовка для реальной строки:**

```
┌─────────────────────────────────────────────────────────────┐
│ t_xmin: 12345  (Иван создал строку)                          │
│ t_xmax: 0      (Никто не удалял)                             │
│ t_ctid: (0,1)  (Строка лежит на странице 0, смещение 1)     │
│ t_infomask: HAS_COLUMNS | HAS_INT_FIELDS                     │
│ t_hoff: 32     (Данные начинаются с 32 байта)               │
├─────────────────────────────────────────────────────────────┤
│ Битовая карта NULL: 00000000 (ни одна колонка не NULL)      │
├─────────────────────────────────────────────────────────────┤
│ Колонка 1 (id): 1 (INT, 4 байта)                            │
│ Колонка 2 (name): "Иван" (VARCHAR, 5 байт)                  │
│ Колонка 3 (email): "ivan@mail.ru" (VARCHAR, 14 байт)        │
└─────────────────────────────────────────────────────────────┘
```

## Часть 3. Как PostgreSQL находит строку? (От SQL до байта на диске)

### Полная диаграмма поиска строки

```
SQL: SELECT * FROM users WHERE id = 42;
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 1. Логический уровень: PostgreSQL знает, что нужно          │
│    найти строку в таблице "users"                           │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 2. Системный каталог (pg_class):                            │
│    SELECT relfilenode FROM pg_class WHERE relname='users'   │
│    → relfilenode = 16386                                    │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 3. Понимает, что файл лежит в:                              │
│    /var/lib/postgresql/16/main/base/16384/16386            │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 4. Открывает файл (через open() системный вызов)           │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 5. Использует индекс (или сканирует таблицу), чтобы         │
│    узнать, на какой странице (блоке) лежит строка 42       │
│    (например, на странице 5)                               │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 6. Вычисляет смещение:                                      │
│    offset = page_number × 8192                              │
│    offset = 5 × 8192 = 40960                                │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 7. Shared Buffers: Проверяет, есть ли страница 5 в кэше?   │
│    - Если есть → читает из RAM (микросекунды)               │
│    - Если нет → читает с диска (lseek + read, миллисекунды) │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 8. Загружает страницу в память, получает указатель на неё   │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 9. Читает Line Pointer по номеру строки внутри страницы    │
│    (если строка — это 2-я строка на странице)              │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 10. Получает смещение строки и её длину                     │
│     Читает HeapTuple заголовок (23 байта)                   │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 11. Проверяет видимость (MVCC):                             │
│     - t_xmin: активна ли транзакция, вставившая строку?     │
│     - t_xmax: не удалена ли строка другой транзакцией?      │
│     Если видима → читаем данные                             │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 12. Пропускает NULL bitmap (если есть)                      │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 13. Читает значения колонок (с учётом типов данных)         │
│     - INT → 4 байта                                         │
│     - VARCHAR → сначала длину (1-4 байта), потом данные     │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 14. Формирует строку в формате TupleTableSlot               │
│     и отправляет клиенту                                    │
└─────────────────────────────────────────────────────────────┘
```

## Часть 4. Физические файлы сопровождения

Каждая таблица имеет не один, а несколько файлов:

### 4.1. Главный файл данных (Main Fork)

```
16386    - файл с данными (обычно)
```

### 4.2. Карта свободного пространства (FSM — Free Space Map)

```
16386_fsm   - файл "карта свободного пространства"
```

**Что это:** Быстро подсказывает, на какой странице есть свободное место для новой строки. Без этого файла INSERT искал бы свободное место перебором всех страниц.

```sql
-- Посмотреть FSM
SELECT * FROM pg_freespace('users');
```

**Пример вывода:**
```
 blkno | avail 
-------+-------
     0 |  2048
     1 |  4096
     2 |     0
```

### 4.3. Карта видимости (VM — Visibility Map)

```
16386_vm    - файл "карта видимости"
```

**Что это:** Отмечает страницы, на которых все строки видны всем транзакциям. Используется для оптимизации:
- `VACUUM` — пропускает "чистые" страницы
- Index-Only Scan — может не проверять таблицу, если страница в VM

```sql
-- Посмотреть VM
SELECT * FROM pg_visibility_map('users');
```

**Пример вывода:**
```
 blkno | all_visible | all_frozen 
-------+-------------+------------
     0 | t           | t
     1 | f           | f
     2 | t           | f
```

### 4.4. TOAST (The Oversized-Attribute Storage Technique)

Если строка становится больше 2 КБ (примерно), PostgreSQL использует **TOAST**:

```
16386         - основная таблица (хранит только ссылки)
16386_toast   - TOAST-таблица (хранит большие данные)
16386_toast_idx - индекс TOAST-таблицы
```

**Когда срабатывает TOAST:**
- `TEXT` длиной 10 КБ
- `JSONB` с большим документом
- `BYTEA` (бинарные данные)

**Как это работает:**
1. Основная таблица хранит только указатель (20 байт)
2. Большие данные разбиваются на чанки по 2 КБ
3. Чанки хранятся в TOAST-таблице

## Часть 5. Практические упражнения

### Упражнение 1: Найти физический файл таблицы

```sql
-- 1. Создайте тестовую таблицу
CREATE TABLE test_physical (
    id SERIAL PRIMARY KEY,
    data TEXT
);

-- 2. Найдите её relfilenode
SELECT relname, relfilenode, relpages 
FROM pg_class 
WHERE relname = 'test_physical';

-- 3. Вставьте данные
INSERT INTO test_physical (data) 
SELECT generate_series(1, 10000), 'Тестовые данные';

-- 4. Снова проверьте relpages (должно увеличиться)
SELECT relname, relfilenode, relpages 
FROM pg_class 
WHERE relname = 'test_physical';
```

### Упражнение 2: Посмотреть содержимое страницы

```sql
-- Установите расширение для просмотра страниц
CREATE EXTENSION IF NOT EXISTS pageinspect;

-- Посмотреть заголовки страниц таблицы
SELECT * FROM get_raw_page('test_physical', 0);  -- страница 0

-- Посмотреть структуру страницы
SELECT * FROM page_header(get_raw_page('test_physical', 0));

-- Посмотреть указатели на строки
SELECT * FROM heap_page_items(get_raw_page('test_physical', 0));
```

### Упражнение 3: Увидеть FSM и VM в действии

```sql
-- Размер файлов в файловой системе
\! ls -la /var/lib/postgresql/16/main/base/$(SELECT oid FROM pg_database WHERE datname = current_database())/

-- Информация из FSM
SELECT * FROM pg_freespace('test_physical');

-- Информация из VM
CREATE EXTENSION IF NOT EXISTS pg_visibility;
SELECT * FROM pg_visibility_map('test_physical');
```

## Часть 6. Что происходит при UPDATE (физически)

Это важно для понимания того, почему `UPDATE` создаёт мёртвые строки и почему нужен `VACUUM`.

```
Исходная строка: id=1, name='Иван', age=25
                    │
                    ▼
┌─────────────────────────────────────────────────────────────┐
│ Шаг 1: PostgreSQL находит страницу со старой строкой        │
│ (например, страница 0, смещение 1)                          │
└─────────────────────────────────────────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────────────────────────┐
│ Шаг 2: Помечает старую строку как "мёртвую":               │
│        - t_xmax = текущая_транзакция                        │
│        - Старая строка остаётся на месте!                   │
└─────────────────────────────────────────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────────────────────────┐
│ Шаг 3: Находит место для НОВОЙ строки                       │
│        (возможно, на другой странице)                       │
└─────────────────────────────────────────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────────────────────────┐
│ Шаг 4: Вставляет новую строку:                              │
│        - t_xmin = текущая_транзакция                        │
│        - t_xmax = 0                                         │
│        - t_ctid указывает на старую строку                  │
└─────────────────────────────────────────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────────────────────────┐
│ Результат:                                                  │
│ - Старая строка → МЁРТВАЯ (жертва VACUUM)                  │
│ - Новая строка → АКТУАЛЬНАЯ                                 │
│ - Таблица распухла (даже если вы обновили 1 строку)        │
└─────────────────────────────────────────────────────────────┘
```

**Визуализация:**
```
ДО UPDATE:
Страница 0: [СТРОКА 1 (актуальная)]

ПОСЛЕ UPDATE:
Страница 0: [СТРОКА 1 (МЁРТВАЯ, t_xmax = 12345)]
Страница 5: [СТРОКА 1 (новая версия)]
```

## Часть 7. Итоговая таблица: Логика vs Физика

| Аспект | Логическая структура | Физическая структура |
|--------|---------------------|---------------------|
| **Что видит пользователь** | Таблицы, строки, колонки | Файлы, байты, страницы |
| **Единица измерения** | Строка (row) | Страница (page) = 8 КБ |
| **Адресация** | По значению (WHERE id=1) | По файлу + странице + смещению |
| **Удаление данных** | `DELETE FROM table` | Помечается как мёртвая (`t_xmax`) |
| **Обновление** | `UPDATE ... SET ...` | Вставка новой версии + старая метится |
| **Ограничения размера** | Теоретически безлимит | 1 ГБ на файл (сегментация) |
| **Группировка** | По схеме (schema) | По каталогу (directory) |
| **Инструменты** | SQL, psql | `pg_class`, `pageinspect`, ls |

## Заключение

PostgreSQL — это **машина отображения** (mapper) между вашим красивым миром таблиц и суровым миром байтов на диске.

**Логическая структура** дает вам:  
✅ Простоту работы с данными  
✅ SQL-интерфейс  
✅ ACID-транзакции  
✅ Независимость от физического расположения  

**Физическая структура** даёт PostgreSQL:  
✅ Контроль над производительностью  
✅ Эффективное использование памяти (Shared Buffers)  
✅ MVCC (многоверсионность)  
✅ Возможность восстановления после сбоев  

**Золотое правило:** Пока ваш запрос выполняется быстро — не важно, что внутри. Но когда он начинает тормозить, понимание физической структуры становится **ключом к оптимизации**.
