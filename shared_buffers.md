# Shared Buffers в PostgreSQL — полное объяснение

## Что такое Shared Buffers простыми словами

**Shared Buffers** — это область **оперативной памяти**, которую PostgreSQL использует для кэширования данных. Она **разделяется** (shared) между всеми процессами-воркерами PostgreSQL.

### Аналогия из жизни

Представьте **библиотеку**:

| Без Shared Buffers | С Shared Buffers |
|-------------------|------------------|
| Каждый раз, когда нужна книга, вы идёте в книгохранилище (диск) | Самые популярные книги стоят на **открытых полках** (оперативная память) |
| Это **очень медленно** (диск в 1000 раз медленнее памяти) | Это **быстро** — книга всегда под рукой |
| 100 читателей одновременно идут в хранилище | Все 100 читателей берут книги с полок |

**Shared Buffers** — это как «горячие полки» в библиотеке, где лежат самые нужные книги, чтобы не ходить каждый раз в далёкое хранилище.

---

## Зачем нужен Shared Buffers

| Характеристика | Чтение с диска | Чтение из Shared Buffers |
|----------------|----------------|--------------------------|
| Скорость | ~0.5-5 мс | ~0.05-0.1 мкс |
| Разница | **Медленно** | **В 10 000 раз быстрее!** |
| Пример | 100 000 строк → 50 секунд | 100 000 строк → 0.5 секунды |

**Вывод:** Shared Buffers ускоряет запросы в тысячи раз.

## Как устроен Shared Buffers (внутреннее устройство)

### Простая схема

```
┌─────────────────────────────────────────────────────────────┐
│                    Оперативная память (RAM)                  │
│  ┌───────────────────────────────────────────────────────┐  │
│  │              Shared Buffers (по умолчанию ~128 МБ)     │  │
│  │  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐        │  │
│  │  │Page 1│ │Page 2│ │Page 3│ │Page 4│ │ ...  │ ...    │  │
│  │  │8 KB  │ │8 KB  │ │8 KB  │ │8 KB  │ │8 KB  │        │  │
│  │  └──────┘ └──────┘ └──────┘ └──────┘ └──────┘        │  │
│  └───────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
                              ↕
┌─────────────────────────────────────────────────────────────┐
│                        Диск (SSD/HDD)                        │
│  ┌────────────────────────────────────────────────────────┐ │
│  │  Файл данных (таблица users):                          │ │
│  │  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐         │ │
│  │  │Page 1│ │Page 2│ │Page 3│ │Page 4│ │Page 5│ ...     │ │
│  │  └──────┘ └──────┘ └──────┘ └──────┘ └──────┘         │ │
│  └────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘
```

### Каждая страница (page)

PostgreSQL делит данные на страницы размером **8 КБ** (по умолчанию):

```
┌─────────────────────────────────────┐
│           Страница 8 KB             │
├─────────────────────────────────────┤
│ Заголовок (24 байта)                │
│ - xmin (ID создавшей транзакции)    │
│ - xmax (ID удалившей транзакции)    │
├─────────────────────────────────────┤
│ Данные:                             │
│ ┌─────┬─────┬─────┬─────┬─────┐    │
│ │row1 │row2 │row3 │ ... │rowN │    │
│ └─────┴─────┴─────┴─────┴─────┘    │
└─────────────────────────────────────┘
```

Одна страница может содержать несколько строк (обычно 10-200 строк в зависимости от размера строки).

## Детальное объяснение каждого шага

### Шаг 1-3: Подключение и запрос
Клиент подключается к `postmaster`, тот создаёт отдельный процесс-воркер. Клиент отправляет SQL.

### Шаг 4-5: Парсинг и оптимизация
- **Парсинг:** Проверка синтаксиса, создание дерева запроса
- **Оптимизация:** PostgreSQL решает, какой план выполнения самый быстрый (использовать индекс или сканировать всю таблицу)

### Шаг 6: Исполнение — нужна страница
Executor понимает, что для ответа нужна страница данных. Например, таблица `users` — файл на диске, разделённый на страницы по 8 КБ. PostgreSQL вычисляет номер страницы по `id`.

**Как вычисляется номер страницы:**
```sql
-- Упрощённо: страница = (смещение строки) / 8KB
```

### Шаг 7-8: Проверка Shared Buffers
Процесс-воркер вызывает функцию `ReadBuffer(block_num)`. Она:
1. Вычисляет хеш от номера страницы
2. Ищет в хеш-таблице Shared Buffers
3. Возвращает указатель, если страница уже в памяти

### Шаг 9a: КЭШ ПОПАЛ (Cache Hit) — хороший сценарий
```
Страница уже в памяти → мгновенный доступ (микросекунды)
```
**Результат:** Данные читаются из RAM, диск не трогается.

### Шаг 9b-12b: КЭШ НЕ ПОПАЛ (Cache Miss) — медленный сценарий
```
Страницы нет в памяти → нужно читать с диска (миллисекунды)
```
PostgreSQL:
1. Находит свободный буфер (или вытесняет старую страницу по алгоритму)
2. Вызывает системный вызов `read()` для чтения с диска
3. Копирует данные из ядра ОС в Shared Buffers
4. Буфер помечается как "грязный" (dirty), если данные будут изменены

### Шаг 13-14: Поиск строки внутри страницы
После того как страница загружена в память, PostgreSQL ищет нужную строку:
1. Проверяет заголовок страницы (доступность строки)
2. Сканирует указатели на строки (line pointers)
3. Проверяет видимость через `xmin`/`xmax` (MVCC)
4. Возвращает строку клиенту

## Диаграмма последовательности: как изменяются данные (UPDATE)

### Сценарий: `UPDATE users SET name = 'Новое имя' WHERE id = 1`

```
┌────────┐     ┌───────────────┐     ┌─────────────┐     ┌─────────┐
│Клиент  │     │Process-воркер │     │Shared       │     │  WAL    │
│(psql)  │     │               │     │Buffers      │     │ Buffer  │
└───┬────┘     └───────┬───────┘     └──────┬──────┘     └────┬────┘
    │                  │                     │                 │
    │ UPDATE ...       │                     │                 │
    │─────────────────>│                     │                 │
    │                  │                     │                 │
    │                  │ 1. Найти страницу   │                 │
    │                  │    с id=1           │                 │
    │                  │    (ReadBuffer)     │                 │
    │                  │────────────────────>│                 │
    │                  │                     │                 │
    │                  │ 2. Захватить        │                 │
    │                  │    блокировку       │                 │
    │                  │    LockBuffer()     │                 │
    │                  │────────────────────>│                 │
    │                  │                     │                 │
    │                  │ 3. Скопировать      │                 │
    │                  │    старую версию    │                 │
    │                  │    (MVCC)           │                 │
    │                  │                     │                 │
    │                  │ 4. Записать         │                 │
    │                  │    изменение в WAL  │                 │
    │                  │─────────────────────────────────────>│
    │                  │  (журнал предзаписи)│                 │
    │                  │                     │                 │
    │                  │ 5. Пометить страницу│                 │
    │                  │    как "грязную"    │                 │
    │                  │    MarkBufferDirty()│                 │
    │                  │────────────────────>│                 │
    │                  │                     │                 │
    │                  │ 6. Освободить       │                 │
    │                  │    блокировку       │                 │
    │                  │────────────────────>│                 │
    │                  │                     │                 │
    │                  │ 7. Отправить        │                 │
    │                  │    подтверждение    │                 │
    │<─────────────────│                     │                 │
    │                  │                     │                 │
    │         ┌────────┴────────┐            │                 │
    │         │  Фоновые процессы              │                 │
    │         │  - Background Writer          │                 │
    │         │  - Checkpointer               │                 │
    │         └────────┬────────┘            │                 │
    │                  │                     │                 │
    │                  │ 8. Позже (через ~1 сек)                │
    │                  │    Background Writer                   │
    │                  │    сбрасывает грязные страницы на диск │
    │                  │─────────────────────────────────────────> (диск)
    │                  │                     │                 │
```

### Ключевые отличия от чтения:

1. **Блокировка страницы** — пока идёт изменение, другие читатели ждут
2. **MVCC** — создаётся НОВАЯ версия строки, старая остаётся
3. **WAL (Write-Ahead Log)** — сначала пишем в журнал, потом на диск
4. **Грязная страница (Dirty)** — помечаем, что данные изменились
5. **Асинхронная запись** — Background Writer позже сбросит на диск

## Структура Shared Buffers (детальное строение)

```
┌─────────────────────────────────────────────────────────────────┐
│                    Shared Memory Region                          │
├─────────────────────────────────────────────────────────────────┤
│  ┌───────────────────────────────────────────────────────────┐  │
│  │  Buffer Descriptors (массив управления)                   │  │
│  │  ┌──────────┬──────────┬──────────┬──────────────────┐   │  │
│  │  │ tag: 42  │ tag: 15  │ tag: 7   │      ...         │   │  │
│  │  │ dirty: 0 │ dirty: 1 │ dirty: 0 │                   │   │  │
│  │  │ pin: 1   │ pin: 0   │ pin: 3   │                   │   │  │
│  │  │ usage: 2 │ usage: 5 │ usage: 1 │                   │   │  │
│  │  └──────────┴──────────┴──────────┴──────────────────┘   │  │
│  └───────────────────────────────────────────────────────────┘  │
│                              ↓                                   │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │  Buffer Content Area (реальные данные страниц)            │  │
│  │  ┌────────┐ ┌────────┐ ┌────────┐ ┌────────────────────┐ │  │
│  │  │Page 42 │ │Page 15 │ │Page 7  │ │        ...         │ │  │
│  │  │ (users │ │(orders)│ │(продукт│ │                    │ │  │
│  │  │  id=1) │ │        │ │   ы)   │ │                    │ │  │
│  │  └────────┘ └────────┘ └────────┘ └────────────────────┘ │  │
│  └───────────────────────────────────────────────────────────┘  │
│                                                                  │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │  Hash Table (хеш-таблица для быстрого поиска)             │  │
│  │  key: page_42 → index: 0                                   │  │
│  │  key: page_15 → index: 1                                   │  │
│  │  key: page_7  → index: 2                                   │  │
│  └───────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

### Что такое Buffer Descriptor (дескриптор буфера)

Для каждой страницы в памяти хранится служебная информация:

| Поле | Что означает | Значение |
|------|--------------|----------|
| `tag` | Идентификатор страницы (таблица + номер блока) | `(relfilenode=12345, blocknum=42)` |
| `dirty` | Грязная ли страница (изменена, не записана на диск) | 0=чистая, 1=грязная |
| `pin` | Счётчик процессов, которые сейчас используют страницу | >0 значит занята |
| `usage` | Счётчик "популярности" (для алгоритма вытеснения) | 0-5, чем больше, тем дольше живёт |

## Алгоритм вытеснения страниц (Clock Sweep)

Когда Shared Buffers заполнен, PostgreSQL нужно решить, **какую страницу удалить**, чтобы освободить место.

### Упрощённая диаграмма

```
┌─────────────────────────────────────────────────────────┐
│  Clock Sweep Algorithm (аналог "часов")                 │
├─────────────────────────────────────────────────────────┤
│                                                          │
│     ┌─────────────────────────────────────┐            │
│     │  Buffer 1 │ Buffer 2 │ Buffer 3 │ ... │            │
│     │ usage=2   │ usage=0  │ usage=5  │     │            │
│     └─────┬─────┴─────┬─────┴─────┬─────┴─────┘            │
│           │           │           │                        │
│           ▼           ▼           ▼                        │
│     ┌─────────────────────────────────────────┐           │
│     │        Clock Pointer (рука часов)       │           │
│     └─────────────────────────────────────────┘           │
│                       │                                    │
│                       ▼                                    │
│     Шаг 1: Смотрим на usage текущего буфера               │
│     Шаг 2: Если usage > 0 → уменьшаем на 1, идём дальше   │
│     Шаг 3: Если usage = 0 → удаляем эту страницу          │
│     Шаг 4: Повторяем по кругу, пока не найдём usage=0     │
└─────────────────────────────────────────────────────────┘
```

**Пример работы:**
```
Начало: [2] → [0] → [5]
Шаг 1: буфер 1 (usage=2) → уменьшаем до 1, идём дальше
Шаг 2: буфер 2 (usage=0) → НАШЛИ! Удаляем эту страницу
Результат: [1] → [НОВАЯ] → [5]
```

## Как посмотреть статистику Shared Buffers

```sql
-- Глобальная статистика кэша
SELECT 
    heap_blks_read,      -- страниц прочитано с диска (Cache Miss)
    heap_blks_hit,       -- страниц прочитано из кэша (Cache Hit)
    round(100 * heap_blks_hit / (heap_blks_hit + heap_blks_read), 2) AS hit_ratio
FROM pg_statio_user_tables
WHERE relname = 'users';

-- Hit ratio должно быть >= 99% для хорошей производительности
```

```sql
-- Размер Shared Buffers
SHOW shared_buffers;

-- Как изменить (требует перезапуска)
ALTER SYSTEM SET shared_buffers = '1GB';
-- Затем перезапустить: sudo systemctl restart postgresql
```

## Настройка Shared Buffers (практические рекомендации)

| Размер ОЗУ | Рекомендуемый shared_buffers | Формула |
|------------|------------------------------|---------|
| 1-2 ГБ | 128-256 МБ | ~15% от RAM |
| 4 ГБ | 1 ГБ | 25% от RAM |
| 8 ГБ | 2 ГБ | 25% от RAM |
| 16 ГБ | 4 ГБ | 25% от RAM |
| 32+ ГБ | 8-10 ГБ | Не более 10-12 ГБ |

**Важное правило:** Shared Buffers не должен быть слишком большим (не более 25-30% ОЗУ), так как:
1. ОС тоже нужна память для дискового кэша
2. У каждого процесса-воркера своя память (work_mem)
3. Слишком большой shared_buffers замедляет восстановление после сбоя

## Итог: что нужно запомнить

| Понятие | Объяснение |
|---------|------------|
| **Что это** | Кэш страниц данных в оперативной памяти |
| **Размер страницы** | 8 КБ |
| **По умолчанию** | 128 МБ |
| **Рекомендуемый размер** | 25% от ОЗУ (но не более 10 ГБ) |
| **Cache Hit** | Данные уже в памяти (быстро) |
| **Cache Miss** | Данных нет в памяти (медленно, чтение с диска) |
| **Целевой hit ratio** | >99% |
